# Fix MongoDB Parallel Array Indexing Issue

## Problem
MongoDB is throwing "cannot index parallel arrays [selectedCottages] [selectedRooms]" error when creating bookings, preventing entrance fee bookings from being saved.

## Root Cause
The Booking schema has compound indexes that include both `selectedRooms.id` and `selectedCottages.id` fields. Even with sparse or partial indexes, MongoDB still sees these as parallel arrays when both fields exist in documents (even if empty).

## Current Attempts
- Added sparse indexes
- Used partial filter expressions  
- Tried separate indexes for each array
- All failed with the same parallel array error

## Alternative Solutions

### Solution 1: Remove Problematic Indexes (Recommended)
**Pros:** Simple, immediate fix, maintains basic query performance
**Cons:** May impact complex availability queries slightly

Remove all compound indexes that include array fields:
```javascript
// Keep only these safe indexes:
bookingSchema.index({ hotelId: 1, checkIn: 1, checkOut: 1, status: 1 });
bookingSchema.index({ userId: 1 });
bookingSchema.index({ createdAt: -1 });
bookingSchema.index({ status: 1, createdAt: -1 });
bookingSchema.index({ paymentStatus: 1, createdAt: -1 });
bookingSchema.index({ checkIn: 1, status: 1 });
```

### Solution 2: Restructure Data Model
**Pros:** Eliminates parallel array issue entirely
**Cons:** Requires schema migration, more complex

Combine selectedRooms and selectedCottages into a single array:
```javascript
selectedItems: [{
  id: String,
  name: String,
  type: 'room' | 'cottage',
  pricePerNight: Number,
  // ... other fields
}]
```

### Solution 3: Application-Level Availability Checking
**Pros:** No database indexes needed for availability
**Cons:** May be slower for large datasets

Move availability checking logic to the application layer instead of relying on database indexes for conflict detection.

## Recommended Approach
**Solution 1** - Remove the problematic indexes. The basic indexes provide sufficient performance for most queries, and availability checking can be handled through application logic or simpler database queries.

## Implementation Steps
1. Remove all indexes containing array fields from Booking schema
2. Test booking creation with entrance fee bookings
3. Monitor query performance and add back essential indexes as needed
4. If performance issues arise, implement Solution 2 (data model restructure)

## Expected Outcome
- Entrance fee bookings will save successfully
- Basic booking queries remain performant
- Complex availability checks may need optimization if performance degrades

## Risk Assessment
- **Low Risk:** Removing complex indexes won't break functionality
- **Performance Impact:** Minimal for typical booking loads
- **Fallback:** Can re-add indexes or restructure data if needed
